# v3.0.9 Release Notes

Date: 2026-05-07 (UTC)

## Headline

Two bundles in one release. **Bundle A — reply-tone hard rules**: when you write in Chinese (or any non-English language), AI replies stay in your language without weaving English mid-sentence; governance internal IDs (`§` codes / `D-XXX` / `F-XXX` / `priority #X` / `Track A/B/C` / `Phase #N` / `Reflection Follow-Through Protocol` / invented `Capital_Snake_Case` names) are banned from sentence subjects; ground truth identifiers (file paths, git SHAs, version numbers, commit hashes, code identifiers) stay allowed inline. Choice options now lead with impact / outcome / what it means for your goal — not "which file changed / how it's done / what tech". Two ghost-reference holes patched: header startup inventory was missing `§0a` Layer Separation; bootstrap backup list was missing 4 v3.0.7+ governance files. **Bundle B — work-pool boundary at session endpoints**: handoff prompt is now codified as "seed context only"; SESSION_LOG entry's `Done` / `Changed` / `Files-edited` fields strictly reflect work performed in THIS conversation; cross-session continued tasks record only the increment.

## What you'll feel

- Replies feel like a colleague writing in your language, not a half-translated paragraph with English fragments stuck in. When you switch to Chinese, AI fully follows — no more `要 update 個 architecture` style mid-sentence English.
- Choice prompts make the trade-off obvious in the first half-line — you don't need to read implementation detail to figure out what each option means for your work. Each option's label leads with impact ("any user clones main can immediately pick up the new rule"), not mechanism ("commit + push to main + cut tag").
- Re-installing on a project that already has wizards / templates no longer skips backing up `dev/wizards/playbook.md` / `dev/wizards/README.md` / `dev/templates/spec_template.md` / `dev/templates/runbook_template.md` — the bootstrap backup list now covers them.
- Header startup inventory now includes `§0a` Layer Separation, matching the always-applicable nature of its hard rules.
- Most importantly: **SESSION_LOG no longer drifts every session**. Old INIT.md users reported repeatedly fixing prior-session done items being copy-pasted into the current session record. That drift is now hard-stopped by governance, not relying on AI mental tracking. Closeout writes only what THIS conversation actually did; cross-session continued tasks record the delta, not the cumulative completion.

## What changed

### Bundle A: Reply-tone hard rules

#### A1. `§11a` rule 5 plain-language strengthen

Three sub-changes merged into the existing rule (zero new top-level rules):

- **Inline-English-mixing ban** — When the user writes in a non-English language (e.g. Chinese), AI does not weave English mid-sentence. English appears only for proper nouns, established terms with no clean translation, or as parenthetical traceability tags.
- **Governance internal ID list expanded + banned from sentence subjects** — Explicit list now covers `§` codes / `D-XXX` / `F-XXX` / `priority #X` / `Track A/B/C` / `Phase #N` / `Reflection Follow-Through Protocol` / invented `Capital_Snake_Case` names. Internal IDs may still appear as parenthetical / end-of-line citations for traceability, but cannot carry the sentence's meaning.
- **Ground truth carve-out** — File paths, git SHAs, version numbers, commit hashes, and function / variable / command names from actual code are not internal IDs; they may appear inline as needed.

#### A2. `§11a` rule 2 outcome-first option framing

Hard rule list extended with one new clause: each choice option's label must lead with impact / outcome / what it means for the user's goal — not with mechanism (which file changed, how it's done, what tech / mechanics). The existing "concrete user-experience preview when non-obvious" rule remains as a layered enhancement.

#### A3. Header startup inventory ghost-reference fix

`§0a` Layer Separation added to the `MANDATORY STARTUP — read every session` inventory at file top. The hard rules in `§0a` (lines 36–40) are always-applicable but were missing from inventory.

#### A4. Bootstrap backup list ghost-reference fix

`§5a` step 9 backup list extended with 4 v3.0.7+ governance files: `dev/wizards/playbook.md`, `dev/wizards/README.md`, `dev/templates/spec_template.md`, `dev/templates/runbook_template.md`. Without this fix, re-installing on a project that already has wizards / templates would skip backing them up.

### Bundle B: Work-pool boundary at session endpoints

Root cause addressed: old INIT.md users reported recurring SESSION_LOG drift — every session, prior-session done items copy-pasted into this session's record, requiring manual cleanup at every closeout. Prior governance left work-pool boundary to AI mental tracking, which drifted in practice.

#### B1. `§1` `Work-pool boundary (mandatory)` — startup-side hard rule

The handoff prompt block is *seed context only* — its description of prior-session completed items, changed files, key decisions, and pending tasks is HISTORY, not part of this session's work pool. PLAN must distinguish:
- prior-session record from handoff = read-only context for understanding state;
- this session's actual work pool = derived from the user's current request + files actually opened / edited / created in THIS conversation.

When the user references a "pending task" from the handoff (e.g., "let's pick up Open Priority 2"), that task enters this session's work pool only on user instruction or on AI's explicit re-decision to act on it — and only the work *actually performed in this conversation* counts toward this session's deliverables.

#### B2. `§4` `Session work-pool boundary (mandatory)` — closeout-side hard rule

SESSION_LOG entry's `Completed` / `Changed` / `Done` / `Files this session edited` fields must reflect only work *actually performed in THIS conversation* — files edited, decisions made, tests run, patches applied in this turn-by-turn session. Do not copy-paste last session's record from the handoff prompt; do not list as `Done` items that were already shipped before this session started.

Cross-session continued tasks: record only the increment performed this session (e.g., `continued from prior session: <delta>`), not the cumulative completion.

Open Priorities are regenerated per the existing rule, re-derived from current project state (handoff seed + this session's deltas + recent SESSION_LOG entries), not copy-pasted from the handoff prompt.

`§1` and `§4` cross-reference each other — startup defines what enters the work pool; closeout defines what gets recorded. Hard separation at both endpoints.
