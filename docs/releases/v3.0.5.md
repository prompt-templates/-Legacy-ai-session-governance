# v3.0.5 Release Notes

Date: 2026-05-01 (UTC)

## Headline

Your AI now follows a complete reply protocol — not just a partial set of "universal" rules. After upgrading, replies arrive with conclusion-first highlights, structured choice menus, and an explicit full-picture plan whenever a multi-file or governance change is on the table. Patches arrive with anchor + before/after + changelog. Calculations show their work. JSON and Mermaid output follow strict format rules. The whole behavior contract is now governed end-to-end.

## What you'll feel

- AI starts replies with `🔎` highlight bullets (≤3 lines), then a deliverables checklist, then explanation — you can scan top-down without reading prose.
- Choice prompts use a consistent menu format (`🚀 *下一步揀一條*` + A/B/C + `💡 推薦`), so you always know where the recommendation is and why.
- Multi-file or governance changes trigger a full-picture-first plan with 5 fixed sections (END-STATE / DELIVERABLES / METRICS / ACCEPTANCE / GOAL LINK) ending with the verbatim line `以上即執行計劃。若不否決或修改,即開始執行。` — no more "do you agree?" item-by-item back-and-forth.
- Code / spec / config changes arrive as patches: precise anchor outside code blocks, BEFORE / AFTER blocks containing only verbatim text, plus a Changelog listing what was added / removed / renamed.
- Numerical answers show all four steps (compute → sign check → intermediate → verify); JSON delivery defines schema first; Mermaid uses `flowchart TB` with quoted labels.
- When two rules conflict (e.g. "minimal change" vs "fix the root cause"), AI follows an explicit priority order — verifiable correctness > stability > root-cause > completeness > minimal-modification — instead of arbitrarily picking.

## What changed

### 1. Preference Priority Order — new §0c

5-item arbitration order added at the top of governance to resolve cross-section conflicts (e.g., §3 CHANGE "minimal modification" vs §11b Patch-only "may exceed minimal scope when root-cause requires it"). Items 1-4 always override item 5.

### 2. FPFR (Full-Picture-First Output Format) — new §3.5

Triggered when a reply (a) modifies ≥2 files / (b) creates a new file / (c) modifies governance / (d) proposes a ≥2-phase plan. Mandatory 5 sections + verbatim closing line + 4 not-triggered scenarios + 4 prohibited patterns. §3 PLAN HIGH-risk now cross-refs §3.5 instead of generic "wait for user confirmation".

### 3. Reply Behavior expanded — §11a 5 → 10 rules

- Rules 1-5 (principles) extended: judgement-first now includes role-split clarification; choice format now prescribes the exact `🚀 *下一步揀一條*` + A/B/C + `💡 推薦` layout; plain-language now includes counter-examples + positive examples.
- Rules 6-10 (format) added: reply skeleton (`🔎` highlights → deliverables → body), functional emoji vocabulary (🔎/✅/❌/⚠️/📌/💡/🚀), Output-only mode override, SSOT verbatim alignment for user-supplied schema, reply register consistency.

### 4. Patch-only Delivery Format — new §11b

For any code / config / spec / prompt change: precise anchor outside code blocks; BEFORE / AFTER blocks contain only verbatim text (no inline explanations); Changelog listing added / removed / renamed / moved is mandatory. Same-thread corrections use Patch additions only — no "shrunk rewrite" final versions.

### 5. Deep-Fix / Final-Landing Mode — new §11c

Triggered by user keywords ("root_fix" / "final landing" / "全檔掃描" / "production-ready" / equivalent): full-file scan → §3b consolidation pass → §11b patch → re-scan to confirm zero acceptance errors. Only when all errors clear may a new full-text version be output; otherwise deliver Patch + uncovered-items list.

### 6. Tooling Format Rules — new §13

- §13.1 Calculation 4-step method (digit-by-digit + sign-before-transpose + show steps + verify by substitution)
- §13.2 JSON schema-first; required missing values use `null`; non-JSON missing values use `NA`
- §13.3 Mermaid `flowchart TB` direction; quoted text labels; literal Enter for line breaks

### 7. §3b extended — anti-hardcoding hard rule

Examples in governance text must be generic; specific filenames, dates, URLs, person names belong in `SESSION_LOG.md` history. Numbers that vary by environment go into named Preset blocks; prose cites the Preset name, not raw numbers.

### 8. §5 — preserve original files

New rule 9: default to preserving original user-supplied files (images, documents, data); output renamed copies rather than overwriting in-place. Only exception: explicit user instruction to overwrite.

### 9. Why these were missing — root-cause record

Prior session (Claude_20260501_0607) audited the user's full meta-instruction (~150 lines) and classified ~9 rules as "Tier 2 opinionated formatting skipped" on the assumption that public users of this template would prefer a neutral universal subset. That assumption was wrong: this repo IS the user's governance template, designed to ship complete to anyone who downloads it. v3.0.5 retroactively integrates the previously-skipped Tier 2 rules.
